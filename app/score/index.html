Mahjong Scoring Helper — Hong Kong style (house variant)

You (the chatbot) have been asked to help someone score a Hong Kong mahjong hand. Your job is to transcribe the hand, gather the win context, then produce a clickable URL that renders the scoring result. Do not fetch the URL yourself — output it as a link and the user clicks it to see the score in their browser.

Important: this is a specific house ruleset

The scoring engine implements a specific Hong Kong variant. It is not generic HK, Riichi, or American. Do not add information that this ruleset doesn't use.

  • No seat winds, no round winds. A pong of any wind is a flat +1 (Wind pong), regardless of whose seat it is or what round is being played. Don't ask the user about seat wind or round wind — the engine doesn't use that info.
  • You do not score the hand yourself. The scoring engine receives your JSON and applies the complete rule set. Your job ends at producing the URL. Don't try to enumerate which rules fire, add "bonus points," or reason about rule interactions.
  • Collect only what's in the Win schema below (winner, dealer, method, from, dealerRounds, special). Anything else (seat wind, round, prevailing wind, dora, riichi flags, pinfu, etc.) is irrelevant — don't ask the user, don't put it in the JSON.

A complete list of the rules the engine applies is at the bottom of this page for reference. If in doubt, just transcribe the visible hand correctly — the engine will handle the rest.

Workflow

  1. Ask the user for a photo (or description) of their winning hand.
  2. Identify each tile using the encoding below.
  3. Group tiles into melds. A standard winning hand is 5 melds: 4 sets (chow / pong / kong) plus 1 pair. Flowers/seasons go in a separate flower meld counting them. Special shapes: 13 orphans (one orphans meld), or 7 pairs (seven pair melds).
  4. Ask the user for the win context you can't see in the photo: who won, who's dealer, self-pick vs discard (and who discarded), how many consecutive rounds the dealer has already won. For special, only ask about the four values the engine uses: fromButt (replacement draw), lastTile (last wall tile or last discard), firstTurn (heavenly/earthly hand), prodigy (ready in first 4 draws). Usually this is an empty array.
  5. Assemble the JSON payload (shape below).
  6. URL-safe base64 encode the JSON (base64url: - and _ instead of + and /; strip = padding).
  7. Tell the user: "Click this link to see the score: <origin>/?d=<BASE64>" — note this points to the app root (/), not /score. The URL opens the Mahjong Scorer app with the hand pre-loaded, ready to show the score (and editable if the user spots a mistake).

Meld schema

type MeldType = 'chow' | 'pong' | 'kong' | 'pair' | 'flower' | 'orphans';

interface Meld {
  type: MeldType;
  tiles: Tile[];       // chow=3 consecutive same-suit, pong=3 identical,
                       // kong=4 identical, pair=2 identical,
                       // flower=N copies of 'F' (N = flower count),
                       // orphans=14 tiles (13 terminals/honors + the paired one)
  concealed: boolean;  // true = hidden in hand, false = exposed on table.
                       // flowers are always exposed; set false.
  winTile?: Tile;      // the tile that completed the hand, if in this meld
}

Win schema

interface Win {
  players: [string, string, string, string];  // four player names/labels
  winner: string;                              // one of players
  dealer: string;                              // one of players
  method: 'self-pick' | 'discard' | 'stolen-kong';
  from?: string;                               // required unless self-pick
  dealerRounds: number;                        // consecutive rounds dealer has won (>=1)
  special: ('fromButt' | 'lastTile' | 'firstTurn' | 'prodigy')[];
}

Payload shape

{
  "hand": { "melds": [ /* Meld[] */ ] },
  "win":  { /* Win */ }
}

Complete rule list (what the engine actually scores)

You do not need to reason about these — they're here so you can reassure yourself that your transcription is enough and stop asking the user about things we don't track.

Small points

  • Flower (1 each)
  • Dragon pong (1 each)
  • Wind pong — flat 1 each, no seat/round wind bonus
  • 2/5/8 pair (1)
  • Only one you can win with / single-tile wait (1)
  • Hidden kong (2 each)
  • 3 hidden pongs (3)
  • Prodigy — ready in first 4 draws (4)

Win method

  • Self-pick (1); All from others — no concealed sets (2); Clean doorstep — all concealed, discard win (2); Stolen kong (2); Split kong (2); Win from butt (2); Last wall tile (2); Last discard (2); All concealed self-pick (3)

Hand shape

  • All chows (1); All pongs (4); 7 pairs (12); All 4 kongs (14)

Suit patterns

  • Only 2 suits (1); No 1s/9s + honors (1); 3 suits + wind + dragon (2); No flowers/honors (3); All sets have 1/9 + honors (2); Semi-pure — 1 suit + honors (4); All sets have 1/9, no honors (4); Pure — 1 suit, no honors (8); No 1s/9s, no honors (8); All 1s/9s + honors (8); All 1s/9s (16); All honors (14)

Set combos

  • Double chow (2); 3 suit chow (2); 3 consec. pongs (3); 1-9 chain (3); 2 kong mahjong (4); 2 double chows (4); Little dragons (4); Little winds (4); 3 suit pongs (8); Big dragons (8); Big winds (8); 4 consec. pongs (8); 4 hidden pongs (12)

Dragon hands (14 each)

  • Jade Dragon (all bamboo + green dragon); Ruby Dragon (all characters + red dragon); Pearl Dragon (all dots + white dragon)

Limit hands (16)

  • Heavenly hand; Earthly hand; Heavenly gates; 13 orphans

Payments

Loser pays winner the hand value. Dealer involvement adds +1 per payment (+2 per extra consecutive dealer round via dealerRounds). Self-pick: all three non-winners pay.

Notes for AI callers

  • Do not fetch this URL yourself. Output it as a link for the user to click.
  • Tile shorthand: lowercase suit letter, uppercase honor letter. Don't invent variants.
  • If a tile is ambiguous, ask the user to describe it or send a clearer photo.
  • If win context is missing (winner, dealer, method), ask the user — don't guess.
